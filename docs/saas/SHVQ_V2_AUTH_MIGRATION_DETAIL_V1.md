## 2) 전환 원칙

1. 구버전 사용자 로그인 호환성을 유지하며 점진 전환한다.
2. 인증 전환은 Wave 0(선행)로 처리하고, 기능 Wave보다 먼저 적용한다.
3. 비밀정보는 코드에서 제거하고 `.env`/시크릿 스토어로 분리한다.
4. 인증 실패 보안(레이트리미팅/세션 재발급/쿠키 보안속성)은 기본값으로 강제한다.

---

## 3) 전환 단계

## 3.1 Phase 1 (긴급)

1. 비밀정보 분리
  - `config.php` 하드코딩 값을 `.env` 기반 로딩으로 교체
2. 비밀번호 정책 전환
  - 신규/변경 비밀번호: `password_hash(..., PASSWORD_BCRYPT)`
  - 로그인 검증: `password_verify()`
  - 레거시 평문/MD5 계정은 로그인 성공 시 Bcrypt로 즉시 업그레이드(점진 마이그레이션)
3. CSRF 적용
  - 모든 상태변경 POST/PUT/DELETE API에 CSRF 토큰 검증 적용
4. 자동로그인 쿠키 강화
  - `HttpOnly`, `Secure`, `SameSite` 속성 강제
  - 평문 ID/권한 쿠키 금지, 서버 검증용 난수 토큰으로 대체

## 3.2 Phase 2 (중기)

1. 세션 고정 방어
  - 로그인 성공/권한 상승 시 `session_regenerate_id(true)` 강제
2. 레이트 리미팅
  - 로그인 기준 `5회/5분` 초과 시 지연 또는 임시 잠금
  - 계정+IP 복합 키 기준 적용
3. 토큰 안전성
  - MD5 토큰 전면 중단
  - `random_bytes()` 기반 토큰/nonce로 교체

## 3.3 Phase 3 (장기)

1. OAuth 2.0 통합(내부/외부 연동 계정)
2. 2FA 단계 적용(관리자/민감권한 우선)
3. 키 로테이션 체계 운영(분기 또는 사고 대응 시 즉시)

## 4) 파일/모듈 적용 우선순위

1. 1순위

---

## 5) 수용 기준(DoD)

1. 평문/MD5 비밀번호 경로가 인증 로직에서 제거된다.
2. 모든 상태변경 엔드포인트에 CSRF 검증이 활성화된다.
3. 로그인 성공 시 세션 재발급이 동작한다.
4. 자동로그인 쿠키에서 평문 식별자/권한 정보가 제거된다.
5. 하드코딩 자격증명 0건을 달성한다.

---

## 6) 롤백/비상 계획

1. 인증 오류 급증 시 레거시 검증 분기를 일시 허용하되, 저장은 Bcrypt만 유지한다.
2. CSRF 도입 초기 장애 시 API 단위 feature flag로 단계 해제한다.
3. 레이트 리미팅 오탐 시 임계값을 즉시 상향하고 감사로그로 재분석한다.
